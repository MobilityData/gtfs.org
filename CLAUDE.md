# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

Source for [GTFS.org](https://gtfs.org/), a static documentation site built with [MkDocs](https://www.mkdocs.org/) and the [`mkdocs-materialx`](https://github.com/jaywhj/mkdocs-materialx) theme (a fork/successor of Material for [MkDocs Materials](https://squidfunk.github.io/mkdocs-material/)). There is no application code here: this is content (Markdown), MkDocs configuration (YAML), and theme customizations (Jinja2 HTML templates, CSS, JS).

## Commands

```bash
# One-time setup (installs requirements.txt + mkdocs-materialx)
make setup

# Serve locally with live reload — only en/fr/es/ja get dev servers, on ports 8000-8003
make serve

# Build all 11 languages into generated/
make build

# Remove the generated/ output directory
make clean
```

There is no test suite or linter configured in this repo.

Before building/serving, run `./scripts/i18n-landing-assets-setup.sh` at least once (both `make build` and `make serve` already do this) — it copies `overrides/assets/landing` into each per-language `overrides/<lang>/assets/` directory and copies untranslated English sections (Resources, Governance) into `docs/fr`, `docs/es`, `docs/ja`. Those copied files are gitignored (see below), so they must be regenerated after a fresh clone before those languages will build cleanly.

## Architecture: one MkDocs build per language

This site does **not** use a single multi-language MkDocs build, which is unusual for MkDocs sites (most rely on `mkdocs-static-i18n` to serve every language from one build/config). That approach was tried here and abandoned: the built-in search plugin did not work correctly with `mkdocs-static-i18n` for any language other than English, to the point that the broken non-English search indexes had to be deleted so they wouldn't interfere with the search tool. The fix was to give each language its own fully separate MkDocs config, content tree, and build — the languages are only "joined" back together at the very end, when the per-language HTML outputs are assembled under `generated/` (English at the root, every other language in its own `generated/<lang>` subfolder). Keep this in mind before "simplifying" the multi-config setup back into a single shared config — that's the exact setup that was already tried and didn't work.

Each language has its own independent config and its own build:

- `config/<lang>/mkdocs.yml` — one file per language (`en`, `fr`, `es`, `ja`, `ru`, `ko`, `id`, `de`, `pt`, `zh`, `zh-TW`). Each sets its own `docs_dir` (`docs/<lang>`), `site_dir` (`generated/` for English, `generated/<lang>` for every other language), its own `nav:` tree, and its own translated strings (e.g. cookie consent text).
- `docs/<lang>/` — Markdown content per language. English (`docs/en`) is the canonical/complete source; most other languages only have partial translations (contributed directly via PRs — see `README.md`) and rely on the untranslated-section copy step above for a few unfinished areas (Resources, Governance, Fares examples).
- `overrides/` — shared `custom_dir` for the theme (Jinja2 templates + assets), referenced by every `config/<lang>/mkdocs.yml`. Per-language asset copies under `overrides/<lang>/` are gitignored and regenerated by the setup script — edit the *source* assets under `overrides/assets/`, not the per-language copies.
- When editing anything that should apply to all languages (nav structure changes aside, since nav is duplicated per-config), check whether the change needs to be replicated across all 11 `config/*/mkdocs.yml` files (e.g. `extra_css`/`extra_javascript` lists have drifted between English and the rest — English currently loads `gtfs-tracker.css`/`gtfs-tracker.js` that other languages don't).

## Theme override structure (`overrides/`)

- `overrides/home.html` — custom landing page template (extends `main.html`, pulls per-language copy from `overrides/locales/<lang>.html` via a `t()` macro).
- `overrides/404.html` — custom 404 page (extends `main.html`).
- `overrides/partials/` — small partial template overrides (`footer.html`, `logo.html`).
- `overrides/assets/stylesheets/extra.css` — the main site-wide stylesheet override. Pay close attention to which rules sit inside which `@media` block: several active-nav/layout rules are intentionally split between an unconditional block and a `@media (min-width: 1220px)` block that mirrors the theme's own desktop/mobile breakpoint — a rule added inside the wrong block silently stops applying at the other breakpoint instead of erroring.
- `overrides/assets/javascript/` — small vanilla-JS enhancements (anchor redirects, search path fixes, language-switch URL handling, a service worker). Note `languageSwitchUrls.js` is referenced with that casing in every `mkdocs.yml`, but the file on disk is `languageswitchurls.js` (all lowercase) — this resolves fine on case-insensitive filesystems (macOS/Windows) but is worth being careful with since the deploy target may be case-sensitive.
- `overrides/locales/` — per-language string tables used only by `home.html`.

## Deployment

Pushing to `main` (excluding docs-only-adjacent paths like `README.md`, `.github/**`, `scripts/**` — see `.github/workflows/prod_push_update_to_server.yml`) triggers a GitHub Actions workflow that updates the live production website (runs locally on the server). Pushing to `staging` does the same against a staging server. The site is served through Cloudflare in front of the origin server — Cloudflare aggressively edge-caches static assets (e.g. `extra.css` had a 2-hour `max-age`), so a change that's live on the origin can still appear stale to users/testers until the CDN cache is purged or expires.

## Git workflow notes specific to this repo

- Branch names should include a date suffix, e.g. `fix/short-description/2026-07-20` (see recent branch/PR history for the convention in use).
- When resolving merge conflicts in `overrides/assets/stylesheets/extra.css` between two independent CSS-rule additions, prefer keeping both rules rather than blindly picking one side — GitHub's web conflict editor makes it easy to accidentally drop a closing `}` or silently discard one side's fix when the changes are adjacent but not truly overlapping.
